A2: Materials by Jeffrey Wang (jw8)

I structured my renderer by adding separate environment, mirror, and PBR pipelines, while keeping the original object pipeline for Lambertian materials. My code now loads HDR RGBE lighting environments, creates Vulkan cubemap views, supports mirror/environment materials, applies exposure and selectable tone mapping, precomputes Lambertian irradiance, and generates GGX-prefiltered cubemap levels through the cube utility. I also added texture descriptor support for normal, displacement, albedo, metallic, and roughness maps in the material pipelines. The only feature I did not fully finish is parallax occlusion mapping: displacement textures can be loaded and bound, but I have not yet completed the shader-side view-dependent UV displacement.

Using the Cube Utility

My cube utility takes a vertically stacked RGBE cubemap PNG as input and precomputes lighting lookup cubemaps for the viewer. The input is stored in RGBE format, which encodes high dynamic range radiance using a shared exponent per pixel. In the cube utility, this RGBE data is first decoded on the CPU into linear HDR radiance values.

In my implementation, I stoe the data into VK_FORMAT_E5B9G9R9_UFLOAT_PACK32 or VK_FORMAT_R16G16B16A16_SFLOA

Once uploaded, the cubemap is used to generate precomputed lighting data. The utility computes a Lambertian irradiance map for diffuse lighting and GGX-prefiltered mip levels for specular reflections. The GGX output is written as a sequence of mip images, such as output_ggx.1.png, output_ggx.2.png, etc., which the viewer loads as environment mip levels.

Command-line Arguments (Cube Utility)

The cube utility reads a vertically stacked RGBE cubemap PNG and writes precomputed cubemap data for environment lighting.

  • --debug, --no-debug -- turn Vulkan debug and validation layers on or off.
  • --lambertian out.png -- compute a Lambertian irradiance cubemap and save it to out.png.
  • --ggx out.png -- compute GGX-prefiltered environment cubemap levels and save them as out.1.png through out.N.png.
  • --ggx-levels N -- choose the number of GGX mip levels to generate. By default, this is based on the input cubemap size.

Command-line Arguments (Viewer)

The viewer loads an s72 scene and renders it with the material pipelines, environment lighting, exposure, and tone mapping options.

  • --debug, --no-debug -- turn Vulkan debug and validation layers on or off.
  • --physical-device name -- run on the named physical device if available.
  • --drawing-size w h -- set the window drawing size.
  • --headless -- run without creating a window and read events from stdin.
  • --scene path -- load a scene from a .s72 file.
  • --camera camera -- render the scene from the named camera.
  • --culling none|frustum|BVH -- choose the scene culling mode.
  • --animation loop|play-once|paused -- control scene animation playback.
  • --exposure E -- scale radiance by 2^E before tone mapping.
  • --tone-map linear|ACES|gamma -- choose the tone mapping curve.

Environment Loading and Display A2-env

Environment maps are loaded in Render::Render. The scene file provides the environment source path, then the renderer loads the vertically stacked cubemap PNG using stbi_load. Each texel is stored as RGBE8, so I convert each loaded glm::u8vec4 RGBE pixel on the CPU using rgbe_to_E5B9G9R9. The converted data is uploaded into a Vulkan cubemap using VK_FORMAT_E5B9G9R9_UFLOAT_PACK32, which keeps HDR range while allowing hardware cubemap sampling and filtering.

The cubemap image is created with six array layers and then viewed with VK_IMAGE_VIEW_TYPE_CUBE. I bind this cubemap through the world descriptor set so that the environment, mirror, Lambertian, and PBR pipelines can sample it. The environment pipeline displays the cubemap directly as the background/environment material, while the mirror pipeline reflects the view direction and samples the same environment cubemap for mirror-like reflections.

I also create and read a separate irradiance cubemap for diffuse environment lighting. After the HDR environment cubemap is uploaded, I allocate World_irradiance as a six-layer cubemap image using VK_FORMAT_R16G16B16A16_SFLOAT. I create both a cube view for sampling and a 2D-array storage view for compute shader writes. The Lambertian irradiance compute shader samples the original environment cubemap, integrates incoming light over the hemisphere, and writes the low-frequency diffuse result into World_irradiance. During rendering, Lambertian materials sample this irradiance cubemap using the surface normal direction, instead of sampling the raw environment map directly.

Tone Mapping A2-tone

For tone mapping, I implemented three modes: linear, ACES, and gamma. Linear mode only applies exposure and leaves the color unchanged afterward. Gamma mode applies exposure first, then converts the linear color to display space using a 1/2.2 power curve. ACES mode uses an ACES-fitted filmic curve based on Stephen Hill's implementation, which compresses bright HDR values more smoothly than gamma and helps preserve highlight detail in environment-lit scenes.

Lambertian Diffuse A2-diffuse

I implement the "lambertian" material in the original object pipeline, using Render::ObjectsPipeline and objects.frag. The renderer binds the world descriptor set, transform buffer, and material texture descriptor set, including the albedo, normal, and displacement textures. In the fragment shader, the material samples its albedo texture and computes a world-space surface normal. That normal is then used to sample the precomputed Lambertian irradiance cubemap, so the material receives diffuse environment lighting from the filtered lookup instead of directly sampling the raw environment map.

Normal Mapping A2-normal

Normal mapping is implemented by loading a normal texture for the material and binding it through the material texture descriptor set. In the fragment shader, the normal map is sampled in tangent space, remapped from texture range [0,1] into vector range [-1,1], and transformed into world space using the tangent, bitangent, and normal basis from the mesh. This perturbed normal is then used for Lambertian and PBR lighting instead of the unmodified geometric normal.

The videos below compare the same material with and without normal mapping. With normal mapping enabled, the silhouette of the object stays the same, b ut the lighting changes across the surface, showing that the normal texture is affecting the shading rather than changing the mesh geometry.

The performance impact of normal mapping was small in my test scenes. It adds one extra texture sample and a tangent-space basis transform in the fragment shader, I did see a jump in time spent drawing a material like the mirror, but less due to the texture sampling.

Physically-Based Material A2-pbr

I implement the "pbr" material in Render::PBRPipeline and pbr.frag. The PBR pipeline binds the world/environment descriptor set, transform buffer, and material texture descriptor set. The material texture set includes albedo, normal, displacement, metallic, and roughness textures. In the fragment shader, I sample these textures and evaluate a GGX microfacet BRDF using the surface normal, view direction, roughness, and metallic parameters.

For enviromentmapping, my implementation follows the split-sum specular IBL approximation from Karis/Epic. Diffuse lighting samples the Lambertian irradiance cubemap. Specular lighting samples a GGX-prefiltered environment cubemap using a roughness-dependent mip level, then combines that value with a preintegrated 2D environment BRDF lookup texture.

I implemented the prefiltering integrations with compute shader dispatches. For Lambertian diffuse lighting, after the HDR environment cubemap is uploaded, I bind the original environment cubemap as a sampled image and bind World_irradiance as a storage image. I then dispatch the Lambertian irradiance compute shader, w hich integrates incoming environment radiance over the hemisphere and writes the filtered diffuse result into the irradiance cubemap.

For PBR specular lighting, I use the split-sum approximation. The GGX-prefiltered cubemap is generated by the cube utility with a GGX compute shader. It importance-samples the environment using the GGX distribution and writes progressively blurrier cubemap mip levels for increasing roughness.

The second part of the split sum, the environment BRDF lookup table, is generated at viewer startup. I allocate World_environment_brdf_lut as a floating-point 2D image, bind it as a storage image, and dispatch the BRDF integration compute shader. After the compute pass finishes, I transition the image for shader sampling and bind it through the world descriptor set. During PBR shading, the fragment shader samples World_environment_brdf_lut using NdotV and roughness to combine the prefiltered environment color with the Fresnel/visibility term.
